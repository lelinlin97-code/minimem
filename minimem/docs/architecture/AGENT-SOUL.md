# 分离式 Agent 架构：当所有器官都被拆走，灵魂是什么？

> 本文档记录了一次关于 AI Agent 架构范式的思考。
> 核心问题：Agent 不应该是单体的，它应该是分离式的——但拆到最后，Agent 自己还剩什么？

---

## 1. 为什么 Agent 不应该是单体的

当前主流 Agent 框架（LangChain、AutoGPT、OpenClaw 等）普遍采用单体架构——感知、记忆、规划、执行、安全、学习全部耦合在一个整体中。

这带来了几个根本性的问题：

| 问题 | 表现 |
|---|---|
| **升级困难** | 想换个更好的记忆方案，却发现和推理逻辑缠在一起，牵一发动全身 |
| **重复造轮子** | 每个 Agent 框架都自己写一套记忆、规划、安全检查，质量参差不齐 |
| **无法组合** | 不能把 A 框架的记忆和 B 框架的规划混着用 |
| **全有或全无** | 用户只想要一个好的记忆系统，却被迫接受整个框架的所有设计决策 |

**类比**：这就像早期的智能手机——相机、音乐播放器、计算器全部焊死在系统里。你不能只升级相机模块，必须换整部手机。

## 2. 分离式架构：Agent 可拆分的核心模块

一个完整的 Agent 可以拆解成以下可独立的"认知器官"：

```
┌─────────────────────────────────────────────────────┐
│                    Agent Host（宿主）                 │
│              轻量编排层，只负责调度和路由               │
│                                                      │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐            │
│   │ Memory   │ │Perception│ │ Planner  │            │
│   │ (记忆)   │ │ (感知)   │ │ (规划)   │            │
│   └──────────┘ └──────────┘ └──────────┘            │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐            │
│   │ Identity │ │ Toolbelt │ │ Guardrail│            │
│   │ (人格)   │ │ (工具带) │ │ (护栏)   │            │
│   └──────────┘ └──────────┘ └──────────┘            │
│   ┌──────────┐ ┌──────────┐                          │
│   │ Learner  │ │ Social   │                          │
│   │ (学习)   │ │ (社交)   │                          │
│   └──────────┘ └──────────┘                          │
└─────────────────────────────────────────────────────┘
```

### 2.1 Memory（记忆）— MiniMem 已实现

长期记忆的存储、检索、巩固、遗忘。

- **为什么能独立**：记忆是状态管理，与推理逻辑无关。任何 Agent 都需要记住东西，但"怎么记"不应该和"怎么想"耦合。
- **已验证**：MiniMem 作为独立 MCP Server 运行，已接入多种 Agent。

### 2.2 Perception（感知/理解层）

把原始输入转化为 Agent 能理解的语义表示。

- **能力范围**：NER 实体识别、意图识别、情感分析、多模态理解、上下文窗口管理
- **为什么能独立**：感知是"输入预处理"，不做决策，只负责"看懂了什么"。就像眼睛和视觉皮层——处理视觉信号，不决定走哪条路。
- **独立价值**：不同 Agent 可以复用同一套感知层。

### 2.3 Planner（规划器）

把高层目标分解为可执行的步骤序列。

- **能力范围**：任务分解、依赖分析、回溯重规划、资源估算
- **为什么能独立**：规划是纯逻辑推理，不需要知道谁来执行、用什么工具。就像 GPS 导航——只算路线，不管你开什么车。
- **独立形态**：MCP Server，暴露 `plan(goal, context, constraints)` → 返回 `Step[]`。

### 2.4 Identity（人格/角色系统）

维护 Agent 的"人设"：语气、性格、知识边界、价值观。

- **能力范围**：Persona 管理、语气一致性、价值观对齐、自我认知
- **为什么能独立**：人格是配置+约束，不参与推理过程，只影响输出的"表达方式"。
- **与 Memory 的关系**：Memory 记住"用户是谁"，Identity 定义"我是谁"。

### 2.5 Toolbelt（工具带/能力注册表）

管理 Agent 能使用的所有工具：发现、注册、鉴权、调用、结果解析。

- **能力范围**：工具注册表、工具选择、调用编排、结果归一化、鉴权和权限
- **为什么能独立**：工具管理是"能力目录"，和推理逻辑无关。MCP 协议已在往这个方向走。

### 2.6 Guardrail（护栏/安全层）

安全检查：输入过滤、输出审核、权限控制、合规检查。

- **能力范围**：注入/越狱检测、PII 脱敏、有害内容过滤、审计日志、速率限制
- **为什么能独立**：安全是横切关注点（cross-cutting concern），不属于任何业务模块。就像 Nginx——拦截所有流量做安全检查，不关心后面跑什么服务。

### 2.7 Learner（学习器/自我进化）

从交互历史中学习，优化 Agent 行为。

- **能力范围**：反馈收集、模式发现、Prompt 自优化、偏好学习
- **为什么能独立**：学习是异步离线过程，不需要实时参与对话。
- **与 Memory 的关系**：Memory 是"记住了什么"，Learner 是"从记忆中学到了什么"。

### 2.8 Social（社交/多 Agent 协调）

Agent 之间的通信、协商、任务委托。

- **能力范围**：消息协议、能力发现、协商机制、委托链
- **为什么能独立**：Agent 间通信是基础设施，和单个 Agent 的业务逻辑无关。

## 3. 灵魂之问：拆完之后 Agent 还剩什么？

```
拆走 Memory     → 它不再能记住
拆走 Perception → 它不再能看懂
拆走 Planner    → 它不再能规划
拆走 Toolbelt   → 它不再能行动
拆走 Guardrail  → 它不再有边界
拆走 Identity   → 它不再有人设
拆走 Learner    → 它不再能进化

还剩什么？
```

### 3.1 Agent 的本质是"意志"（Will）

剩下的是一个极薄但极关键的层——**意图循环（Intent Loop）**：

```
              ┌──────────────────────────┐
              │     Agent Core (意志)     │
              │                          │
              │    感知到了什么？          │
              │         ↓                │
              │    我想要什么？           │
              │         ↓                │
              │    我该怎么做？           │
              │         ↓                │
              │    做完了吗？满意吗？      │
              │         ↓                │
              │       (循环)             │
              └──────────────────────────┘
```

### 3.2 人体类比

| 独立模块 | 对应人体 |
|---|---|
| Memory | 海马体（记忆） |
| Perception | 感觉皮层（感知） |
| Planner | 前额叶（规划） |
| Toolbelt | 运动皮层 + 四肢（行动） |
| Guardrail | 道德直觉 + 杏仁核（边界） |
| Identity | 自我概念（人格） |
| Learner | 突触可塑性（学习） |
| **Agent Core** | **脑干 + 丘脑（生命维持 + 信号路由）** |

脑干维持你活着（基本循环不停），丘脑负责路由（把感知信号分发到正确的皮层区域）。它们自己不"思考"，但没有它们，所有高级能力都无法被串联起来。

### 3.3 Agent Core 的四个不可替代职责

**1. 状态驱动（State Machine）**

```
idle → perceiving → thinking → planning → acting → evaluating → idle
```

每个 Agent 的状态流转不同。客服 Agent 是"监听→理解→回复"，编程 Agent 是"读需求→规划→写代码→测试→修复"。这个流转逻辑是 Agent 独有的"性格"。

**2. 意图裁决（Intent Arbitration）**

当多个信号同时出现时，Agent Core 决定优先级：
- 用户发了新消息 + 定时任务到了 + 记忆系统说该做梦了 → 谁优先？
- 这个判断不属于任何子模块，是 Agent 自己的"意志"。

**3. 上下文编织（Context Weaving）**

从 Memory 拿记忆，从 Perception 拿意图，从 Identity 拿人设，从 Guardrail 拿安全边界——**编织成一个 prompt 发给 LLM**。

这个编织过程是 Agent 最核心的"手艺"。同样的原材料，不同的编织方式，产出完全不同的对话质量。

**4. 反馈闭环（Feedback Loop）**

执行完动作后，评估结果并决定通知谁：
- 用户满意了？→ 告诉 Learner
- 产生新知识？→ 告诉 Memory
- 计划要调整？→ 告诉 Planner
- 出现安全风险？→ 告诉 Guardrail

**"告诉谁什么"的决策，是 Agent 自己的判断。**

## 4. 指挥家隐喻

```
Agent = 薄但有主见的指挥家

它不自己演奏任何乐器，
但它决定：
  - 什么时候谁该演奏
  - 以什么节奏和力度
  - 各声部怎么配合
  - 整首曲子的情感走向

没有指挥家，乐团只是一群各自练琴的人。
没有乐团，指挥家只是一个空挥手臂的人。
```

## 5. 终局推演：Agent 变成协议

如果把分离式架构推到极致：

```
单体 Agent (现状):
  [感知 + 记忆 + 规划 + 执行 + 安全 + 学习] = 一个大黑盒

分离式 Agent (目标):
  Agent Host（薄编排层）
    ├── MCP → Memory Server (MiniMem)
    ├── MCP → Perception Server
    ├── MCP → Planner Server
    ├── MCP → Identity Server
    ├── MCP → Toolbelt Server
    ├── MCP → Guardrail Server
    ├── MCP → Learner Server
    └── MCP → Social Server
```

> **分离式架构的终局，是 Agent 变成了一个"协议"而非"程序"。**

每个独立模块定义自己领域的协议，Agent Core 变成一个**符合编排协议的配置文件 + 一小段粘合代码**。

这和 **Unix 哲学**一致——每个程序做好一件事，用管道连起来。只不过这次的"管道"是 MCP 协议，"程序"是各个独立的认知模块。

**MiniMem 是这个生态里的第一个"命令"。**

## 6. 开放问题

- Agent Core 的"上下文编织"能力，是否也可以标准化？还是它天然是每个 Agent 的核心差异化？
- 分离式架构下，延迟怎么控制？每个模块都是独立进程，模块间通信成本如何？
- 如何处理模块间的循环依赖？（比如 Planner 需要查 Memory，Memory 的检索策略又需要 Planner）
- 是否存在一个"最小可行 Agent"——只需要 Core + Memory 就能运行，其他都是渐进增强？
